# HPE Workflow Automation

## Executive Summary

This case study documents the strongest current proof point in the portfolio: a workflow automation system built around Ross Perry's day-to-day motion as a `Lead Generation Representative (LGR)` within Concentrix B2B Sales, supporting the HPE logo. The goal was not to automate selling away. The goal was to reduce preventable admin drag, preserve context across disconnected tools, and make follow-through easier to scale.

## Business Context

The selling motion depended on multiple systems working together:

- Salesforce for opportunity and account context
- RingDNA for outreach execution and activity
- Outlook for communication continuity and stakeholder follow-up

Each system was useful on its own, but the operating experience between them was fragmented. That fragmentation created duplicate entry, inconsistent capture, and more manual effort than the motion should require.

## Problem Statement

The business problem was straightforward:

- too much seller time was spent on mechanical capture
- context was at risk when work moved between tools
- follow-through quality depended too heavily on individual memory
- the operating model was not easily transferable to other programs

## Solution Summary

The response was a dashboard-led workflow automation model backed by prompt-engineered process scaffolds. Instead of treating each platform as an isolated task surface, the system framed them as connected inputs into a shared operating layer.

That shared layer emphasized:

- normalized workflow steps
- prompt frameworks for repeatable actions
- better visibility into what had happened and what needed to happen next
- reusable automation patterns that could be adapted elsewhere

## Measured Outcome

The most important result was operational:

- `30-40%` reduction in manual data-entry time

The broader value was architectural:

- reduced context switching
- clearer workflow continuity
- stronger repeatability
- a pattern worth considering for use in adjacent Concentrix programs

## Why This Matters For A Solutions Architect Role

This project demonstrates solutions-architecture behavior before the title:

- diagnosing friction as a system problem, not a user problem
- designing around business outcomes instead of technical novelty
- connecting prompt engineering to operational process design
- thinking in reusable patterns rather than one-off fixes
